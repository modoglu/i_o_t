3. Ingress Yönlendirme Mantığı (Traffic Control)

ingress.yaml dosyası, trafiği Host-based routing yöntemiyle yönetir:

  • app1.com -> app1-service (Port 80)
  • app2.com -> app2-service (Port 80)
  • Diğer tüm trafik (Default) -> app3-service
Evaluator İpucu: "Nasıl test ederiz?" diye sorarsa: /etc/hosts dosyasına bu domainleri ekleyerek veya curl -H "Host: app1.com" 192.168.56.110 komutuyla kanıtlayabilirsin.

4. Ortam Değişkenleri ve Gizli Kahramanlar

Kodunda kullandığın MESSAGE değişkeni dışında, K3s ve App yönetimi için bilmen gerekenler:

# Uygulama içindeki değişken (YAML'dan)
env:
  - name: MESSAGE
    value: "Hello from appX"

# K3s Kurulum Değişkenleri (Genişletilmiş)
- K3S_KUBECONFIG_MODE: Dosya izinlerini ayarlar (Senin 644 yaptığın yer).
- K3S_DATASTORE_ENDPOINT: Eğer etcd veya harici SQL kullanacaksan.
- K3S_AGENT_TOKEN: Worker'lar için kritik güvenlik anahtarı.

5. Defans (Evaluation) Soruları

S: Neden Port 8080'den 80'e yönlendirme yaptın?
C: Uygulama (container) güvenli olmayan yüksek portlarda (8080) çalışır. Service katmanı bunu standart HTTP portuna (80) çekerek Ingress için uyumlu hale getirir.

S: app2 neden 3 replika?
C: Load Balancing (Yük dengeleme) özelliğini göstermek için. Bir pod çökse dahi diğer ikisi trafiği karşılamaya devam eder.

Kritik Uyarı: server_setup.sh içinde until sudo kubectl get nodes döngüsü çok önemli. Çünkü K3s'in API server'ının ayağa kalkması birkaç saniye sürer. Bu döngü olmadan YAML dosyalarını apply etmeye çalışırsan "Connection Refused" hatası alırsın.

Raporu sunduktan sonra: "Dilerseniz bir pod'u silip K3s'in onu nasıl otomatik olarak tekrar ayağa kaldırdığını (Self-healing) gösterebilirim," diyerek sunumu bitirebilirsin.

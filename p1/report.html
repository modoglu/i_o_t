2. Kurulum Betikleri ve Mantıksal Akış

2.1. Master Setup (server_setup.sh)

[cite_start]

Kurulumda kullanılan INSTALL_K3S_EXEC değişkeni, cluster'ın davranışını belirleyen en kritik noktadır: [cite: 1]

export INSTALL_K3S_EXEC="--node-ip=192.168.56.110 --flannel-iface=eth1 --write-kubeconfig-mode 644 --disable traefik"
    [cite_start]
  • --flannel-iface=eth1: Flannel CNI'ın Vagrant'ın iç ağı üzerinden haberleşmesini zorunlu kılar. [cite: 1]
    • [cite_start]
  • --write-kubeconfig-mode 644: Kubeconfig dosyasının yetkilerini düzenleyerek sudo gereksinimini azaltır. [cite: 1]
    • [cite_start]
  • --disable traefik: Özel bir Ingress çözümü kullanılacağı için varsayılanı kapatır. [cite: 1]

2.2. Worker Setup (worker_setup.sh)

Worker, Master'ın ürettiği node-token dosyasına ihtiyaç duyar. [cite_start]Bu dosya /vagrant paylaşımlı dizini üzerinden aktarılır: [cite: 2]

until [ -r /vagrant/node-token ]; do sleep 1; done
[cite_start]

Bu döngü, Master kurulumu tamamlanıp token oluşana kadar Worker'ın beklemesini sağlar (Race Condition Prevention). [cite: 2]

4. Evaluator Savunma (Defans) Rehberi

Soru: IP adreslerini neden manuel (hardcoded) verdin?

Cevap: Cluster içindeki servislerin (özellikle API Server ve DNS) birbirini bulabilmesi için sabit IP gereklidir. Dinamik IP kullanımı, bir restart durumunda tüm cluster iletişiminin kopmasına neden olur.

Soru: alias k='kubectl' neden önemli?

[cite_start]

Cevap: .bashrc dosyasına eklenen bu alias, operasyonel hızı artırır ve 42 evaluatörüne terminale ve araçlara olan hakimiyetimizi gösterir. [cite: 1, 2]

Soru: set -e kullanmasaydın ne olurdu?

Cevap: Kurulum sırasında bir paket (örneğin curl) inmezse script hata vermeden devam ederdi ve biz k3s'in neden kurulmadığını anlamakta zorlanırdık. [cite_start]set -e hata anında durmayı garanti eder. [cite: 1, 2]
